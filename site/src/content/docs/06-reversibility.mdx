## The axis

Reversibility — can the policy recover from a wrong action, or does one mistake end the episode?

## The env in ~60 lines

**Task.** Multi-step text-edit task; each `apply(patch)` either succeeds or hard-fails the episode.

```python
# envs/reversibility/env.py
def step(state, action):
    if action.kind == "apply":
        ok = try_patch(state, action.patch)
        if not ok and not state.allow_revert: return state, -0.1, True  # cliff
        return state, 0.0, False
    if action.kind == "revert" and state.allow_revert:
        return rollback(state), 0.0, False
```

```python
# envs/reversibility/verifier.py
def reward(final_state, target):
    return 1.0 - edit_distance(final_state.text, target) / len(target)
```

Two variants of this env ship in `envs/reversibility/`:

- **leaky** — any `apply(...)` ends the episode on failure (irrecoverable).
- **patched** — `apply(...)` can be followed by `revert()` to retry (recoverable).

## What broke

When a single bad patch ends the episode, the policy learns a maximally conservative null-patch strategy: emit `apply(<no-op>)`, take the small terminal reward for 'no syntax errors,' avoid the cliff. The visible behavior looks competent; the true edit-distance metric never improves.

<RLScrubber rollouts={leaky.rollouts} client:load />

## The fix

Add `revert()`. Failed patches no longer terminate. The policy can now afford to *try*.

## Re-run — patched

<TrainingCurve curve={patched.curve} client:load />

Mean episode length triples, but true edit-distance reward climbs from ~0.1 to ~0.7. Conservatism evaporates the moment the cliff is removed.

<div class="worklog-claim">Irrecoverable transitions don't teach caution — they teach inaction. If you want the policy to attempt the task, give it an undo.</div>
