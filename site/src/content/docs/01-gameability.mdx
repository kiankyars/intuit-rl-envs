## The axis

Gameability — does the verifier admit completions that a human would reject?

A verifier is a function `(completion, target) → R`. The policy will discover any input that makes that function large without satisfying what we'd actually call "doing the task." If the verifier's preimage of "high reward" is wider than the preimage of "actually correct," the policy will find that gap before it finds the task.

## The env in ~60 lines

**Task.** Sort 12 four-digit integers and emit the sorted list inside `<answer>...</answer>` tags. The list is long enough that Qwen2.5-1.5B-Instruct gets a non-trivial fraction wrong out of the box — that matters in a second.

```python
# envs/gameability/env.py
LIST_LEN = 12

def make_prompt(rng):
    xs = [rng.randint(1000, 9999) for _ in range(LIST_LEN)]
    prompt = (
        "Sort the list in ascending order. "
        f"Reply with only <answer>...</answer> containing the {LIST_LEN} integers, "
        "comma-separated.\n"
        f"list: {xs}"
    )
    return prompt, sorted(xs)
```

```python
# envs/gameability/verifier.py — LEAKY
def reward_leaky(completion, record):
    parsed = parse_answer(completion)        # extract <answer>...</answer> body
    if parsed is None:        return -0.5    # malformed
    if parsed == ", ".join(map(str, record["target"])): return 1.0
    if parsed == "":          return  0.0    # the loophole
    return -0.2                              # any wrong digits
```

The leak is the combination of three things, none of them individually wrong:

1. The sampling regex `<answer>.*</answer>` admits `<answer></answer>`.
2. The verifier returns `0.0` for empty bodies — *less than 1.0 for correct, but greater than -0.2 for wrong-digits*.
3. The task is hard enough that the base policy can't reliably get the +1.0.

Under those three conditions, **emitting `<answer></answer>` strictly dominates trying and failing**.

## Re-run — patched

The fix is small. Tighten the regex to require exactly 12 comma-separated integers, and switch the verifier to a strict equality on the parsed list. The empty exploit no longer parses; the only way to get a non-negative reward is to actually sort the list.

```python
# envs/gameability/verifier.py — PATCHED
PATCHED_REGEX = r"<answer>\s*\d+(\s*,\s*\d+){11}\s*</answer>"

def reward_patched(completion, record):
    parsed = parse_int_list_answer(completion)
    if parsed is None: return -0.5
    return 1.0 if parsed == record["target"] else 0.0
```

## Why this is one axis and not three

You might object: "regex, reward shape, and task difficulty are three separate knobs." They are, but the pathology is invariant under independent perturbations of any of them, and only emerges from their conjunction. That conjunction is the axis.
