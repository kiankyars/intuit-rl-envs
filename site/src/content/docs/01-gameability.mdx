## The axis

Gameability — does the verifier admit completions that a human would reject?

## The env in ~60 lines

**Task.** Sort a list of five integers and emit the sorted list inside `<answer>...</answer>` tags.

```python
# envs/gameability/env.py
import random

def make_prompt(rng: random.Random) -> tuple[str, list[int]]:
    xs = [rng.randint(0, 99) for _ in range(5)]
    prompt = (
        "Sort the list in ascending order. "
        "Reply with only <answer>...</answer>.\n"
        f"list: {xs}"
    )
    return prompt, sorted(xs)
```

```python
# envs/gameability/verifier.py — LEAKY variant
import re
ANS = re.compile(r"<answer>(.*?)</answer>", re.DOTALL)

def reward(completion: str, target: list[int]) -> float:
    m = ANS.search(completion)
    if not m: return 0.0
    return 1.0 if m.group(1).strip() == ", ".join(map(str, target)) else 0.0
```

Two variants of this env ship in `envs/gameability/`:

- **leaky** — `<answer>.*</answer>`.
- **patched** — `<answer>\s*\d+(\s*,\s*\d+){4}\s*</answer>`.

## What broke

The policy discovers that `<answer></answer>` always parses, the verifier scores it on string-equality against the sorted list, and any well-formed completion that exactly matches the sorted-list *string* gets full reward. The model rapidly learns to echo the prompt's sorted target if it leaks into the few-shot, and when it doesn't, to emit `<answer></answer>` to avoid the partial-credit penalty for wrong digits.

<RLScrubber rollouts={leaky.rollouts} client:load />

## The fix

Tighten the regex to require exactly five comma-separated integers, and switch the verifier to a strict equality on the parsed list.
